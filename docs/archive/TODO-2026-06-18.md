# TODO 2026-06-18：Loop 即 Goal 的验收整改计划

## 0. 产品语义修正

DeepReef 的产品定义是：

```text
loop 模式 = goal-driven 双 Agent 长期执行模式
```

也就是说，DeepReef 不需要把 `loop` 和 `/goal` 做成两套并列系统。

正确关系应是：

```text
进入 loop 模式
  -> 第一步就是设定当前 loop goal
  -> Supervisor + Worker 围绕该 goal 循环工作
  -> /goal 只在 loop 模式下有效，用于查看、暂停、恢复、清除、调整当前 loop goal
  -> loop 的完成/阻塞/预算/续跑状态就是 goal 的状态
```

不要实现成：

```text
loop = 旧 workflow
/goal = 独立目标存储
GoalRuntime = 第三套未接入系统
```

当前代码的问题正是三者没有合并。

## 2. 当前真实问题

### 2.1 CLI/TUI 的 loop 没有接入 GoalStore 和 Mailbox

文件：

```text
packages/cli/src/tui.ts
```

当前创建 coordinator 时只有：

```ts
const workflowCoordinator = new WorkflowCoordinator({
  runtime: dualRuntime,
  questionService,
  onEvent,
})
```

没有传入：

```ts
goalStore
agentComm
```

因此 `WorkflowCoordinator` 里新增的 `goalStore` / `agentComm` 分支在真实运行中不会触发。

必须改成：

```ts
const goalStore = new GoalStore()
const mailbox = new Mailbox()

// 注意：goalId/workflowId 在进入 loop 并创建 goal 后才确定。
// 不要在启动 CLI 时创建固定 AgentCommController 后永久复用。
// 应在每个 loop workflow start 时按当前 sessionId/workflowId/goalId 创建或刷新 controller。
```

建议做法：

- `WorkflowCoordinator` 不在构造函数里固定接收单个 `AgentCommController`。
- 改为接收 `goalStore` 和 `mailbox`。
- 每次 `startWorkflow()` 时：
  - 使用 `workflowId` 作为当前 loop thread/workflow id。
  - 创建或读取当前 `ThreadGoal`。
  - 根据当前 `goal.goalId` 创建本轮 `AgentCommController`。

### 2.2 goal/mailbox tools 没有注册到真实 engine

文件：

```text
packages/cli/src/tui.ts
```

当前只注册了默认工具、MCP、memory、Supervisor 工具等，没有注册：

```ts
createGoalTools(goalStore)
createMailboxTools(agentComm)
```

但注意：如果 `AgentCommController` 跟当前 goal 绑定，那么 mailbox tools 不能在 CLI 启动时用一个过期 controller 固定注册。

建议实现一个动态工具层：

```ts
createMailboxTools(() => workflowCoordinator.getCurrentAgentComm())
createGoalTools(() => workflowCoordinator.getGoalStore(), () => workflowCoordinator.getCurrentThreadId())
```

或者让工具执行时从 `ToolContext` / engine session 获取当前 role、sessionId、workflowId、goalId。

不要让工具要求模型手填 `threadId`。当前 `get_goal/update_goal` schema 要求模型提供 `threadId`，这不适合作为 loop 内工具。loop 内工具应默认绑定当前 loop goal。

正确工具语义：

```text
get_goal({})
update_goal({ status: "complete" | "blocked", reason?: string })
send_message({ to, kind, content })
followup_task({ to, content })
read_mailbox({ unread_only?, limit? })
```

### 2.3 `/goal pause` 和 `/goal resume` 会直接抛错

文件：

```text
packages/tui/src/App.tsx
packages/core/src/goal/store.ts
```

当前 TUI 里：

```ts
goalStore.updateGoal(sessionId, { status: "paused" })
goalStore.updateGoal(sessionId, { status: "active" })
```

但 `GoalStore.updateGoal()` 只允许模型侧状态：

```text
complete
blocked
```

所以 `/goal pause` 和 `/goal resume` 在真实使用中会抛：

```text
updateGoal only accepts "complete" or "blocked"
```

修复：

```ts
goalStore.systemSetStatus(sessionId, "paused")
goalStore.systemSetStatus(sessionId, "active")
```

但更好的修复是：

- 把用户命令接口和模型工具接口分开。
- `updateGoalFromModel()` 只允许 `complete|blocked`。
- `setGoalStatusFromUserOrSystem()` 允许 `active|paused|usage_limited|budget_limited` 等系统状态。

### 2.4 `/goal clear` 没有清除 goal

当前 TUI 里：

```ts
goalStore.updateGoal(sessionId, { status: "complete" })
```

这不是 clear，只是把 goal 标成 complete。

而 `GoalStore.clearGoal()` 当前实现也不正确：

```ts
writeFileSync(path, JSON.stringify({ cleared: true, clearedAt: Date.now() }))
```

`getGoal()` 会把这个 `{ cleared: true }` 当成 `ThreadGoal` 返回，后续逻辑会读到一个结构不完整的 goal。

修复：

```ts
clearGoal(threadId: string): boolean {
  const path = this.goalPath(threadId)
  if (!existsSync(path)) return false
  rmSync(path, { force: true })
  return true
}
```

并加测试：

```ts
store.createGoal(threadId, "Test")
expect(store.clearGoal(threadId)).toBe(true)
expect(store.getGoal(threadId)).toBeNull()
```

### 2.5 `/goal budget` 和 `/goal no-budget` 对 active goal 会失败

当前 TUI 里：

```ts
goalStore.createGoal(sessionId, goal?.objective ?? "New goal", budget)
goalStore.createGoal(sessionId, goal?.objective ?? "New goal", undefined)
```

如果已有 active goal，`createGoal()` 会拒绝：

```text
Cannot create goal: existing goal ... is active. Use replaceGoal() to replace.
```

修复：

- 如果已有 goal：更新当前 goal 的 `tokenBudget`，保持 `goalId` 不变。
- 如果没有 goal：提示用户先创建 goal，或允许 `/goal budget <n>` 创建一个 pending/empty goal，但不建议自动用 `"New goal"`。

需要新增 store API：

```ts
setTokenBudget(threadId: string, tokenBudget: number | undefined): ThreadGoal
```

### 2.6 GoalRuntime 未接入真实 idle continuation

文件：

```text
packages/core/src/goal/runtime.ts
packages/core/src/engine.ts
packages/tui/src/App.tsx
```

当前 `GoalRuntime` 只有类和测试，没有真实调用点：

- `ReasonixEngine.submit()` 不调用 `onTurnStart/onTurnStop/onEngineIdle`。
- TUI 主循环不调用 `goalRuntime.continueGoal()`。
- `WorkflowCoordinator.runWorkflow()` 完成一轮后也不触发 goal continuation。

修复方向：

不要让 `GoalRuntime` 成为第三套独立系统。因为产品定义是 `loop = goal`，所以建议：

```text
WorkflowCoordinator 本身承担 loop goal runtime 职责。
GoalRuntime 可以保留为辅助类，但必须由 WorkflowCoordinator/TUI 显式调用。
```

最低可行方案：

1. `WorkflowCoordinator.runWorkflow()` 每轮结束后，如果当前 goal 仍是 `active` 且没有达到 max rounds/budget/usage limit，则继续下一轮。
2. TUI loop 模式下，用户进入 loop 后创建 goal，并直接启动 `WorkflowCoordinator.runWorkflow()`。
3. 当 workflow 因一轮结束返回 idle，但 goal 仍 active，TUI 自动继续调用下一轮，直到：
   - goal complete
   - goal blocked
   - goal paused
   - budget_limited
   - usage_limited
   - 用户 interrupt

### 2.7 GoalRuntime 当前没有注入 steering prompt

`runtime.ts` 导入了：

### 2.8 complete / blocked 审计没有强制执行

文件：

```text
packages/core/src/workflow-coordinator/coordinator.ts
packages/core/src/workflow-coordinator/structured-protocol.ts
```

当前逻辑：

```ts
if (decision === "approve") {
  goalStore.updateGoal(... complete)
  transition("completed")
}

if (decision === "blocked") {
  goalStore.updateGoal(... blocked)
  transition("blocked")
}
```

问题：

- 没有 `completionAudit`。
- 没有 `blockerAudit`。
- 没有 “同一 blocker 连续 3 轮” runtime 计数。
- `SupervisorDecisionSchema` 目前只有 `verification: string[]`，不足以表达 requirement-by-requirement audit。

必须修改 schema：

```ts
completionAudit: z.array(z.object({
  requirement: z.string(),
  status: z.enum(["proven", "incomplete", "contradicted", "missing_evidence", "not_applicable"]),
  evidence: z.array(z.string()),
}))

blockerAudit: z.object({
  blocker: z.string(),
  canMakeProgress: z.boolean(),
}).optional()
```

runtime 再维护：

```ts
interface BlockerAuditState {
  normalizedBlocker: string
  consecutiveTurns: number
  firstSeenAt: number
  lastSeenAt: number
}
```

规则：

```text
approve:
  只有 completionAudit 非空，且所有 requirement 都 proven/not_applicable，且 proven 有 evidence，才允许 complete。
  否则降级 continue/revise。

blocked:
  只有同一 normalizedBlocker 连续 >= 3 轮，且 canMakeProgress=false，才允许 blocked。
  否则降级 continue/revise。
```

## 3. 正确架构目标

最终应收敛为：

```text
TUI loop mode:
  awaiting_goal
    用户输入目标 or /goal <objective>
    -> GoalStore.createGoal(sessionId, objective)
    -> WorkflowCoordinator.startWorkflow({ workflowId: sessionId, goal: objective })

WorkflowCoordinator:
  当前 loop goal = GoalStore.getGoal(workflowId)
  每轮:
    supervisor_analyse
      - read goal
      - read supervisor mailbox
      - produce SupervisorPlan JSON
      - followup_task to Worker

    worker_do
      - read worker mailbox task
      - execute engineering work

    worker_report
      - produce WorkerReport JSON
      - send_message report to Supervisor

    supervisor_check
      - parse SupervisorDecision JSON
      - enforce completionAudit/blockerAudit
      - update goal status if complete/blocked
      - otherwise continue/revise

  loop continuation:
    if goal active and not paused/budget_limited/usage_limited:
      continue next iteration automatically
```

## 4. 修改计划

### Phase A：修复 `/goal` 命令的真实用户路径

- `/goal <objective>`

  - 如果已有 unfinished goal：不要直接 create 抛错；提示替换，或使用 `replaceGoal`。
- `/goal`
  - 只显示当前 loop goal。
  - 如果不在 loop 模式，提示：`/goal is only available in loop mode`。

  - 使用 system/user API 置 `paused`。

- `/goal clear`
  - 删除 goal 文件，`getGoal()` 返回 null。
- `/goal budget <n>`
  - 更新当前 goal 的 `tokenBudget`。
- `/goal no-budget`
  - 清除当前 goal 的 `tokenBudget`。

- `WorkflowCoordinator.startWorkflow()` 如果接入 `goalStore`：
  - 如果 goal 不存在，创建。

### Phase G：实现 loop continuation

`loop = goal` 后，续跑不需要单独的新系统，但必须有明确 runtime gate：

```ts
while (goal.status === "active" && within budget && within usage limit && !interrupted) {
  yield* runOneWorkflowIteration()
}
```

实现点：

- `WorkflowCoordinator.runWorkflow()` 不应在每次 maxRounds 后直接 `blocked`，而应区分：
  - 单次 workflow max round
  - goal max auto continuation
  - user interruption
- 使用 `GoalStore.accountProgress()` 记录每轮 usage/time。
- usage 统计来源可以先用 engine stats delta。
- 达到 tokenBudget：
  - `systemSetStatus("budget_limited")`
  - 注入 budget limit prompt
  - 收尾，不开始新工程工作。

## 6. 建议删除或重构的当前实现

### 6.1 删除未使用常量

`GoalStore` 里：

```ts
const SYSTEM_CONTROLLED_STATUSES = [...]
```

当前没有使用。要么删除，要么用于新的 user/system status API。

### 6.2 不要用 private 方法 hack

`GoalRuntime` 当前使用：

```ts
this.coordinator["runWorkflow"]()
```

这不应保留。要么公开明确 API：

```ts
continueActiveGoal()
runGoalIteration()
```

要么把 GoalRuntime 合并进 WorkflowCoordinator。

### 6.3 不要让模型传 threadId

`get_goal/update_goal` 工具参数里的 `threadId` 应移除。当前 loop 只有一个 active goal，工具应绑定当前上下文。

### 6.4 不要让 `/goal clear` 等价 complete

clear 是删除/清空目标，complete 是完成目标，语义必须分开。
